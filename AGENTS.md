# AGENTS.md — AI Development Conventions & Truths

This document serves as the source of truth and development conventions for any AI Coding Assistant working on the **CIAO! Sicily — Kitchen Hub** project. Always read and adhere to these guidelines to ensure consistency, cleanliness, and correctness.

---

## 1. Core Architectural Truths

- **Framework:** The project is a full-stack **TanStack Start** application (powered by React 19, TanStack Router, and Nitro SSR).
- **Routings:** All route definitions are under `src/routes/`. The routing tree is auto-generated by TanStack Router (outputting to `src/routeTree.gen.ts`). Do **NOT** edit `routeTree.gen.ts` manually.
- **Server Functions:** Write server-side handlers using `createServerFn` (see [example.functions.ts](file:///Users/wisleyaguiar/Projetos/ciao-sicily-kitchen/src/lib/api/example.functions.ts)).
- **Server-Only Files:** Any module containing server-only functions, database drivers, or secrets must have the `.server.ts` extension. This prevents Vite from accidentally bundling server logic into the client code.

---

## 2. Asset Management (CRITICAL)

- **Local Images Only:** We do **NOT** use Lovable's external CDN-based `.asset.json` files. All images must be standard format files (`.png`, `.jpg`, etc.) located in [src/assets/](file:///Users/wisleyaguiar/Projetos/ciao-sicily-kitchen/src/assets).
- **Asset Imports:** Import images directly as JS modules in React:
  ```typescript
  import myImage from "@/assets/my-image.png";
  ```
- **Direct Usage:** Vite resolves these imports to standard URL strings. Do **NOT** use the `.url` suffix when rendering images:
  ```tsx
  /* CORRECT */
  <img src={myImage} alt="Brand" />

  /* INCORRECT (Will break rendering) */
  <img src={myImage.url} alt="Brand" />
  ```

---

## 3. Styling & Custom Theme (Tailwind CSS v4)

- **No Config File:** The project uses **Tailwind CSS v4**. There is no `tailwind.config.js` or `postcss.config.js`.
- **CSS Rules:** All styling tokens, custom theme configurations, custom variant definitions, and keyframe animations must be configured inside [styles.css](file:///Users/wisleyaguiar/Projetos/ciao-sicily-kitchen/src/styles.css) under `@theme inline`.
- **Color Format:** All brand colors are strictly declared in the **OKLCH** color space (e.g. `oklch(0.58 0.22 27)`). Ensure any new colors added to the theme follow the OKLCH specification for consistent rendering.
- **Brand Patterns:**
  - **Checker Bands:** Use `.checker-band` for diagonal Italian conic patterns.
  - **Signature Text:** Use `.ciao-script` (which uses the *Permanent Marker* font family rotated at `-3deg`) to render handwriting elements.

---

## 4. Modularity & Component Conventions

- **Clean Route Files:** The main route file [index.tsx](file:///Users/wisleyaguiar/Projetos/ciao-sicily-kitchen/src/routes/index.tsx) acts exclusively as a coordinator/layout container.
- **Section Modularity:** Layout sections are modularized under [src/components/sections/](file:///Users/wisleyaguiar/Projetos/ciao-sicily-kitchen/src/components/sections). Any new section, visual addition, or major content block must be created as a separate file in this folder.
- **Modular UI Elements:** Reusable visual UI elements (buttons, dialogs, drawers, accordions) must remain in [src/components/ui/](file:///Users/wisleyaguiar/Projetos/ciao-sicily-kitchen/src/components/ui) and be imported as needed.

## 5. SEO & Structured Data Markup

- **Metatags:** Manage page meta titles, OpenGraph tags, descriptions, and links inside route head declarations using TanStack Start's `head` API (e.g., in [index.tsx](file:///Users/wisleyaguiar/Projetos/ciao-sicily-kitchen/src/routes/index.tsx)).
- **Dynamic FAQ Schema:** The [FaqSection.tsx](file:///Users/wisleyaguiar/Projetos/ciao-sicily-kitchen/src/components/sections/FaqSection.tsx) renders a dynamic JSON-LD structured data script (`FAQPage`) generated from the `faqItems` array. If you modify the questions/answers, the JSON-LD updates automatically. Ensure this script is never deleted or broken during updates.

## 6. How to Build & Verify

Before submitting any code changes, verify correctness:
1. Run local dev server to check runtime:
   ```bash
   npm run dev
   ```
2. Run production compilation to test TypeScript types and Vite bundle outputs:
   ```bash
   npm run build
   ```

## 7. Netlify Deployment Conventions

- **Vite Config:** Nitro packaging must be enabled by setting `nitro: true` in [vite.config.ts](file:///Users/wisleyaguiar/Projetos/ciao-sicily-kitchen/vite.config.ts). This is required for server function compilation and routing.
- **Build Target:** For local builds intended for Netlify deployment, prefix the build with `NITRO_PRESET=netlify` to output files to `.netlify/functions-internal` and `.output/public`.
- **Deploy Command:** Deploy changes locally using the Netlify CLI:
  ```bash
  NITRO_PRESET=netlify npm run build
  netlify deploy --prod
  ```

## Browser Verification with Playwright MCP

Whenever browser-based verification is required, always use the Playwright MCP tools and follow these rules:

1. Before opening the browser, verify that the local development server is running and reachable.
2. Prefer `http://127.0.0.1:<port>` instead of `http://localhost:<port>` to avoid hostname or loopback resolution issues.
3. Open the target page using Playwright MCP and wait until the page reaches a stable state before inspecting it.
4. Check for:

   * visible layout and responsive behavior;
   * JavaScript runtime errors;
   * failed network requests;
   * broken images or missing assets;
   * incorrect spacing, alignment, overflow, or overlapping elements;
   * accessibility issues that are visible or reported by the browser;
   * differences between the implementation and the provided design or requirements.
5. Test the relevant viewport sizes when the task affects responsive behavior.
6. Capture screenshots whenever visual validation is relevant.
7. Do not claim that a page was visually verified unless the page was successfully opened and inspected through Playwright MCP.
8. If the browser context fails to initialize, do not repeatedly retry the same command without investigation. First:

   * confirm that the server is reachable at `127.0.0.1`;
   * confirm that the expected port is correct;
   * check whether the Playwright browser or CDP connection is available;
   * retry once after correcting the detected issue.
9. If Playwright MCP remains unavailable, clearly report the exact error and state that browser verification could not be completed. Do not fabricate screenshots, visual findings, or successful validation.
10. After completing browser verification, provide a concise report containing:

    * the URL and viewport tested;
    * the checks performed;
    * any issues found;
    * any fixes applied;
    * whether the final verification passed.
